Overview — What Is Broken And Why

After reviewing all relevant source files, three specific issues are
causing WINSTON to feel robotic. They are listed here in priority order
— fix them in sequence, do not jump ahead.

The AI brain (openai.ts, docent-persona.ts) is working correctly. The
persona is being passed, conversation history is being sent. The
problems are in the greeting layer, a persona routing bug, and the TTS
voice quality.

Fix 1 — Replace Hardcoded Greeting With AI-Generated Greeting

  ----- -------------------------------------------------------------------
  BUG   In ChatInterface.tsx and PersistentChatInterface.tsx, the welcome
        message is generated by generateGreeting() — a template engine that
        stitches together random string fragments. This is not WINSTON. It
        is a Mad Lib. WINSTON's first impression must be written by the AI
        every single time.

  ----- -------------------------------------------------------------------

1.1 — Create a New API Route: /api/chat/greeting

Create a new file at: src/app/api/chat/greeting/route.ts

This route accepts a POST request with the following body:

  -----------------------------------------------------------------------
  { artworkId: string, museumId: string, visitorName: string | null }

  -----------------------------------------------------------------------

The route must:

1.  Load the artwork from the database exactly as the main /api/chat
    route does

2.  Build a ChatContext with the artwork data and visitor name

3.  Call createChatCompletion with the following special system
    instruction appended to the persona:

  -----------------------------------------------------------------------
  This is the FIRST message the visitor will see. Do not repeat the
  persona rules back. Write a single, natural opening — 2 to 3 sentences
  maximum. You are standing with the visitor in front of this specific
  artwork right now. React to the artwork as if seeing it together for
  the first time. Use the visitor's name once if you have it. Do not say
  'Hello' or 'Welcome'. Do not start with 'I'. Start with something that
  pulls them into the painting immediately.

  -----------------------------------------------------------------------

4.  Return the generated greeting as JSON: { greeting: string }

  ------ -------------------------------------------------------------------
  NOTE   This route should use the non-streaming path (stream: false) and
         max_tokens: 120. Greetings should be short and punchy. The voice:
         true flag should be set so the voice persona is used — short
         sentences, spoken rhythm.

  ------ -------------------------------------------------------------------

In the loadArtwork useEffect, replace this line:

The same fix applies here. PersistentChatInterface does not currently

  ------ -------------------------------------------------------------------
  NOTE   Do not delete greeting-generator.ts yet. Keep it in place as a

Fix 2 — Fix The Voice Persona Routing Bug

  ----- -------------------------------------------------------------------
  BUG   In openai.ts, the buildSystemPrompt call uses 'voice || stream' as
        the voice flag. This means ALL streaming requests use the voice
        persona — including text chat. The voice persona limits sentences
        to 15 words, so text conversations are being unnaturally truncated.
        This is a single character fix but it has a significant impact on
        response quality.

  ----- -------------------------------------------------------------------

2.1 — The Exact Bug

In src/lib/ai/openai.ts, find this line inside createChatCompletion:

  -----------------------------------------------------------------------
  const systemPrompt = buildSystemPrompt(groundingContext,
  context.artwork, voice || stream, context.visitorName);

  -----------------------------------------------------------------------

The third argument is: voice || stream

This evaluates to true whenever stream is true, which it is for all
voice tour calls. But it is ALSO true for any streaming text call, which
is incorrect.

2.2 — The Fix

Change the third argument from voice || stream to simply voice:

  -----------------------------------------------------------------------
  const systemPrompt = buildSystemPrompt(groundingContext,
  context.artwork, voice, context.visitorName);

  -----------------------------------------------------------------------

Then verify that in route.ts, the voice: true flag is only being passed
when the request is an actual voice tour request. Check the stream call
in PersistentChatInterface.tsx:

  -----------------------------------------------------------------------
  body: JSON.stringify({ message: content, artworkId, museumId:
  actualMuseumId, visitorName: visitorName || null, stream: useStreaming,
  conversationHistory, // voice flag should only be true when
  isVoiceTourActive is true voice: session.isVoiceTourActive, }),

  -----------------------------------------------------------------------

If the voice field is not currently being sent in the request body, add
it. Then in route.ts, extract it and pass it through to
createChatCompletion:

  -----------------------------------------------------------------------
  const { message, artworkId, museumId, visitorName, stream: useStream,
  conversationHistory, voice = false } = await req.json();

  -----------------------------------------------------------------------

  -----------------------------------------------------------------------
  // Pass voice flag through to createChatCompletion const response =
  await createChatCompletion(context, { model: 'gpt-4o-mini', maxTokens:
  useStream ? 500 : 600, groundingTokenLimit: 1000, historyTokenLimit:
  1500, stream: useStream, voice: voice, // ← this was missing });

  -----------------------------------------------------------------------

  ------ -------------------------------------------------------------------
  NOTE   After this fix: text chat gets the full DOCENT_PERSONA with rich
         responses. Voice tour gets the DOCENT_VOICE_PERSONA with short
         spoken sentences. They were both getting voice persona before,
         which explains why text responses felt clipped and robotic.

  ------ -------------------------------------------------------------------

Fix 3 — Replace Web Speech TTS With OpenAI TTS

This is the fix that will have the most dramatic perceived impact. The
Windows/Chrome Web Speech API voices (Microsoft David, Zira) sound like
a GPS system. OpenAI's TTS API uses the same API key already in use — no
new accounts, no new billing setup. Cost per museum visit is under
$0.05.

The microphone and speech recognition side (SpeechRecognition API) stays
exactly as it is — free and working. Only the speaking output is being
replaced.

3.1 — Create a New API Route: /api/tts

Create a new file at: src/app/api/tts/route.ts

This is a simple proxy route between the client and OpenAI TTS API:

  -----------------------------------------------------------------------
  import { NextRequest } from 'next/server'; import OpenAI from 'openai';
  const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
  export async function POST(req: NextRequest) { const { text } = await
  req.json(); if (!text?.trim()) { return new Response('Missing text', {
  status: 400 }); } const mp3 = await openai.audio.speech.create({ model:
  'tts-1', voice: 'onyx', input: text, speed: 0.95, }); const buffer =
  Buffer.from(await mp3.arrayBuffer()); return new Response(buffer, {
  headers: { 'Content-Type': 'audio/mpeg', 'Cache-Control': 'no-cache',
  }, }); }

  -----------------------------------------------------------------------

  ------ -------------------------------------------------------------------
  NOTE   Voice 'onyx' is deep, calm, and authoritative — the best fit for
         WINSTON's character. If the museum wants a different tone,
         alternatives are: 'echo' (neutral, clear), 'fable' (warm,
         British-adjacent), 'nova' (energetic). Do not use 'shimmer' or
         'alloy' — they read as too feminine or too robotic for this use
         case.

  ------ -------------------------------------------------------------------

3.2 — Update WinstonVoiceManager.ts — Replace speak() Method

The current speak() method in WinstonVoiceManager.ts uses
window.speechSynthesis. Replace the entire speak() method body with an
OpenAI TTS fetch call. Keep the method signature identical — nothing
else in the class needs to change.

Replace the speak() method with:

  -----------------------------------------------------------------------
  async speak(text: string): Promise<void> { console.log(`[Voice]
  Speaking via OpenAI TTS: "${text.substring(0, 50)}..."`);
  this.stopSpeaking(); this.setMode('speaking'); return new Promise(async
  (resolve, reject) => { try { const response = await fetch('/api/tts', {
  method: 'POST', headers: { 'Content-Type': 'application/json' }, body:
  JSON.stringify({ text }), }); if (!response.ok) throw new Error(`TTS
  API error: ${response.status}`); const audioBlob = await
  response.blob(); const audioUrl = URL.createObjectURL(audioBlob); const
  audio = new Audio(audioUrl); this.currentAudio = audio; audio.onended =
  () => { URL.revokeObjectURL(audioUrl); this.currentAudio = null;
  console.log('[Voice] TTS completed'); resolve(); }; audio.onerror =
  (err) => { URL.revokeObjectURL(audioUrl); this.currentAudio = null;
  console.error('[Voice] Audio playback error:', err); reject(err); };
  await audio.play(); } catch (error) { console.error('[Voice] TTS fetch
  failed:', error); reject(error); } }); }

  -----------------------------------------------------------------------

3.3 — Update stopSpeaking() Method

The current stopSpeaking() calls this.synthesis.cancel(). Update it to
also stop the Audio element:

  -----------------------------------------------------------------------
  public stopSpeaking(): void { // Stop OpenAI TTS audio if playing if
  (this.currentAudio) { this.currentAudio.pause();
  this.currentAudio.currentTime = 0; this.currentAudio = null; } // Keep
  the synthesis cancel as fallback if (this.synthesis?.speaking) {
  this.synthesis.cancel(); } }

  -----------------------------------------------------------------------

3.5 — Handle The Firefox Consideration

The reason we switched from Chrome to Firefox earlier was the mic/audio
conflict. OpenAI TTS uses the HTML Audio API (not Web Speech synthesis),
which does NOT have the same conflict. This means Chrome should work
correctly again with this fix in place. Test on both browsers after
implementing.
